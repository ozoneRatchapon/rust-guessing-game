# Phase 3: MagicBlock VRF — Free, Fast Trustless On-Chain Randomness

## Overview

This phase implements a **trustless guessing game** on Solana using MagicBlock's Ephemeral VRF (Verifiable Random Function). It is a **separate** Anchor program (`phase3-magicblock-vrf`) from Phase 1 and Phase 2 — all three programs coexist on devnet with different Program IDs.

**How it works:**

1. The admin **initializes** a game, marking it as awaiting VRF randomness.
2. The admin **requests randomness** via CPI to the MagicBlock VRF program.
3. The MagicBlock oracle generates randomness off-chain and **calls back** into our program with the result.
4. The callback **derives the secret number** (1–100) from the VRF bytes and stores a blake3 hash as an audit trail.
5. Players **guess** the number. The program responds with "too small", "too big", or "correct".

**Trust model:** Nobody — including the admin — knows the secret number before the VRF callback. The secret is derived from a cryptographic randomness proof generated by the MagicBlock oracle using Curve25519 Ristretto group operations and Schnorr-like signatures (RFC 9381). This eliminates the Phase 1 limitation where the admin chose (and therefore knew) the answer.

**Key differences from Phase 2 (Switchboard VRF):**

- **Free** — no SOL cost for randomness requests (MagicBlock subsidizes oracle costs).
- **Fast** — <1 second delivery. The oracle processes within a single slot, versus ~3 seconds for Switchboard.
- **Callback pattern** — the VRF program CPIs *back* into our program with randomness, instead of us reading a settled account.

---

## Why a Separate Program (Not Replace Phase 2)

Phase 3 lives in its own program alongside Phase 1 and Phase 2, not as a replacement:

1. **Phase 1 and Phase 2 are already deployed and demo-able forever.** They work, they're tested, and they teach different patterns (commit-reveal vs. Switchboard VRF).
2. **Phase 3 has different instructions.** It adds `request_randomness` and `consume_randomness` — a callback-based flow unique to MagicBlock's VRF architecture. The account struct is different too (`GameV3` with VRF-specific fields).
3. **Better for teaching.** Students compare three approaches side by side: trust-on-admin → Switchboard VRF → MagicBlock VRF.
4. **Git tag `phase2-demo` preserves the Phase 2 snapshot.** You can always checkout an earlier version.

All three programs run on devnet simultaneously with separate Program IDs. A frontend can integrate any or all of them.

## Architecture

### Phase 1 Flow (Trust-on-Admin)

```
initialize(secret_number) → reveal(secret_number) → guess(guess)
        |                          |                       |
  Admin picks number         Admin reveals number     Player guesses
  Stores blake3 hash         Verifies hash            Program compares
  Admin knows secret         Secret now public        Emits event
```

### Phase 2 Flow (Switchboard VRF — ~3 second oracle wait)

```
initialize → [wait ~3s for oracle] → settle_random → guess(guess)
     |                                       |                |
 Admin creates game                    Admin settles VRF   Player guesses
 Commits to Switchboard randomness     Derives secret      Program compares
 Nobody knows secret                   Secret now on-chain  Emits event
```

### Phase 3 Flow (MagicBlock VRF — <1 second, free)

```
initialize → request_randomness → [oracle <1s] → consume_randomness → guess(guess)
     |              |                                    |                  |
 Admin creates    Admin CPIs to                     VRF program CPIs    Player guesses
 GameV3 PDA       MagicBlock VRF                    back with bytes     Program compares
 vrf_pending=true  Submits request                  Derives secret      Emits event
                  Nobody knows secret               blake3 audit trail
```

**Order matters:** `initialize` → `request_randomness` → (oracle responds) → `consume_randomness` → `guess`. You cannot guess before the VRF callback is consumed. The oracle response (<1 second) is dramatically faster than Switchboard's ~3 second wait.

## MagicBlock VRF Callback Pattern

MagicBlock's Ephemeral VRF uses a **callback pattern** — the key architectural difference from Switchboard's settle-based approach:

1. **Request** (`request_randomness`) — Our program CPIs to the MagicBlock VRF program, submitting a randomness request. The request includes:
   - A `caller_seed` (derived from the admin's `client_seed`)
   - The callback program ID (our program)
   - The callback discriminator (the `consume_randomness` instruction)
   - Callback account metas and args (empty in our case)

   ```rust
   // We CPI to VRF program with request data
   // The VRF program queues the request for the oracle
   ```

2. **Oracle generates randomness** off-chain — The MagicBlock oracle observes the request and generates a provably random value using Curve25519 Ristretto group operations with Schnorr-like signatures (RFC 9381). This takes <1 second.

3. **Callback** (`consume_randomness`) — The VRF program CPIs **back** into our program with the 32 bytes of randomness. The VRF program signs with its identity PDA (`VRF_PROGRAM_IDENTITY`), which we verify as the only authorized caller.

   ```rust
   // VRF program calls our consume_randomness(randomness: [u8; 32])
   // We verify the signer is VRF_PROGRAM_IDENTITY
   // We derive: secret = random_u8_with_range(&randomness, 1, 100)
   ```

**Why this is trustless:** The randomness is derived from cryptographic primitives (RFC 9381) that are mathematically provable. The VRF program identity PDA ensures only the legitimate oracle can deliver randomness. Even the admin cannot influence the result because the oracle uses slot hashes as entropy.

### SDK Compatibility Issue

The `ephemeral-vrf-sdk` v0.2.3 is **incompatible** with Anchor 1.0.1 due to a `Pubkey` type mismatch — the SDK uses `solana-program` 2.x `Pubkey` while Anchor 1.0.1 uses `solana-pubkey` 3.x. We worked around this by:

1. **Inlining VRF constants** directly in `constants.rs` (program ID, queue, identity, seeds).
2. **Manually building CPI instructions** in `request_randomness.rs` using `Instruction` structs and `invoke_signed`.
3. **Inlining the serialization structs** (`RequestRandomnessData`, `InlinedAccountMeta`) with Borsh.

This is documented with comments throughout the code. When the SDK resolves the compatibility issue, the manual CPI code can be replaced with SDK calls.

## Instructions

### initialize(client_seed)

**Purpose:** Admin creates a new game. The `vrf_request_pending` flag is set to `true`, signaling that the game is awaiting a VRF randomness request.

**Account validation:**

- `game` — initialized as a PDA with seeds `[b"game_v3", admin.key()]`, funded by the admin.
- `admin` — must be a signer and must pay for account rent.
- `system_program` — required for account creation.

**What gets stored:**

```rust
game.admin = admin.key();
game.secret_hash = [0u8; 32];      // placeholder, filled after VRF callback
game.secret_number = 0;             // NOT known yet
game.is_revealed = false;           // VRF not consumed
game.attempts = 0;
game.max_tries = MAX_TRIES;         // 10
game.is_finished = false;
game.bump = bump;
game.vrf_request_pending = true;    // awaiting VRF request
```

The `client_seed` parameter is passed through to the VRF request (see `request_randomness` below). It provides additional entropy but does not let the admin control the outcome — the oracle's cryptographic proof ensures unpredictability.

---

### request_randomness(client_seed)

**Purpose:** Admin CPIs to the MagicBlock VRF program, submitting a randomness request. The VRF program will call back to our `consume_randomness` instruction when the oracle responds.

**Account validation:**

- `program_identity` — the VRF program identity PDA, derived from seeds `[b"identity"]` with the VRF program as the seeds::program. Used as a signing PDA for the CPI.
- `oracle_queue` — the MagicBlock default oracle queue (`Cuj97ggrhhidhbu39TijNVqE74xvKJ69gDervRUXAxGh`). Validated by address.
- `slot_hashes` — the SlotHashes sysvar, needed by the VRF program for seed derivation.
- `vrf_program` — the MagicBlock VRF program (`Vrf1RNUjXmQGjmQrQLvJHs9SNkvDJEsRVFPkfSQUwGz`).
- `payer` — must be a signer, pays for the CPI.

**How the CPI is built:**

Since we can't use the `ephemeral-vrf-sdk`, we manually construct the CPI instruction:

```rust
// 1. Build the request data (borsh-serialized)
let request_data = RequestRandomnessData {
    caller_seed,                              // from client_seed, padded to 32 bytes
    callback_program_id: crate::ID,           // our program
    callback_discriminator: CONSUME_RANDOMNESS_DISCRIMINATOR,
    callback_accounts_metas: vec![],          // no extra accounts needed
    callback_args: vec![],                    // randomness passed by VRF program
};

// 2. Prepend the VRF discriminator
let mut data = REQUEST_RANDOMNESS_DISCRIMINATOR.to_vec();
request_data.serialize(&mut data)?;

// 3. Build the Instruction with correct account metas
let ix = Instruction {
    program_id: vrf_program_id,
    accounts: vec![
        AccountMeta::new(payer, true),
        AccountMeta::new_readonly(program_identity, true),  // signs via PDA seeds
        AccountMeta::new(oracle_queue, false),
        AccountMeta::new_readonly(system_program::ID, false),
        AccountMeta::new_readonly(slot_hashes_id, false),
    ],
    data,
};

// 4. Invoke with PDA signer seeds
invoke_signed(&ix, &[...], &[&[IDENTITY_SEED, &[identity_bump]]])?;
```

**Security checks:**

1. **Admin-only:** The signer must match the game's stored admin.

2. **VRF request pending:** The game must be in a state where a VRF request is expected.

---

### consume_randomness(randomness)

**Purpose:** Callback instruction invoked by the MagicBlock VRF program. The VRF oracle generates randomness off-chain and the VRF program CPIs back to this instruction with the 32-byte result. This is the core trustless mechanism.

**Account validation:**

- `vrf_program_identity` — must be a signer AND must match the known VRF identity address (`9irBy75QS2BN81FUgXuHcjqceJJRuc9oDkAe8TKVvvAw`). This ensures only the legitimate VRF oracle can invoke this callback.
- `game` — mutable (we're writing the secret number and hash).

**Security checks (in order):**

1. **VRF identity verification:** The signer must be the VRF program identity PDA.

   ```rust
   #[account(address = VRF_PROGRAM_IDENTITY.parse::<Pubkey>().unwrap())]
   pub vrf_program_identity: Signer<'info>,
   ```

2. **Not already revealed:** Prevents consuming randomness twice.

   ```rust
   require!(!game.is_revealed, GameError::AlreadyRevealed);
   ```

3. **Game not finished:** Cannot consume randomness on a completed game.

   ```rust
   require!(!game.is_finished, GameError::GameFinished);
   ```

**Deriving the secret number:**

```rust
let secret_number = random_u8_with_range(&randomness, 1, 100);
let hash = blake3::hash(&secret_number.to_le_bytes());

game.secret_number = secret_number;
game.secret_hash = *hash.as_bytes();
game.is_revealed = true;
game.vrf_request_pending = false;
```

The `random_u8_with_range` function uses **rejection sampling** to avoid modulo bias:

```rust
fn random_u8_with_range(bytes: &[u8; 32], min_value: u8, max_value: u8) -> u8 {
    let range = (max_value - min_value + 1) as u16;  // 100
    let threshold = (256 / range * range) as u8;      // 200

    for &b in bytes.iter().rev() {
        if b < threshold {
            return min_value + (b % range as u8);
        }
    }
    // Fallback (slight bias, but rare — only when all 32 bytes >= threshold)
    min_value + (bytes[31] % range as u8)
}
```

This scans the 32 VRF bytes looking for a value that maps uniformly into [1, 100]. Values ≥ 200 are rejected to avoid the modulo bias that would otherwise favor numbers 1–56. The blake3 hash is stored as an **audit trail** — anyone can verify the secret number matches the on-chain hash after the game ends.

### guess(guess)

**Purpose:** Any player can submit a guess. The program compares it to the secret and emits an event. Identical logic to Phase 1 and Phase 2.

**Account validation:**

- `game` — mutable (we're tracking attempts).
- `player` — must be a signer (any wallet, no restriction).

**Security checks (in order):**

1. **Game not finished:** Cannot guess on a completed game.

2. **Secret revealed (VRF consumed):** Cannot guess before randomness is consumed.

3. **Guess range valid:** Guess must be in [1, 100].

4. **Attempts remaining:** Player has not exceeded `max_tries`.

**Comparison logic:**

```rust
match guess.cmp(&game.secret_number) {
    std::cmp::Ordering::Equal => {
        game.is_finished = true;
        emit!(GuessCorrect { guess, attempts: game.attempts });
    }
    std::cmp::Ordering::Less => {
        emit!(GuessTooSmall { guess, attempts: game.attempts });
    }
    std::cmp::Ordering::Greater => {
        emit!(GuessTooBig { guess, attempts: game.attempts });
    }
}
```

**Auto-finish:** After the guess, if `attempts >= max_tries` and the game isn't already finished, the game ends automatically.

---

### close_game()

**Purpose:** Admin closes the game account, recovering the rent-exempt lamports back to their wallet. Identical to Phase 1 and Phase 2.

**Account validation:**

- `game` — mutable, closed via Anchor's `close` constraint. PDA seeds `[b"game_v3", admin.key()]` with bump.
- `admin` — must be a signer. Receives the recovered rent lamports.

**When to use:** After a game is finished and you want to start a new one at the same PDA.

## Security Checks

The VRF callback flow introduces unique attack vectors. Here are the security checks that prevent them:

| Check | Where | What it prevents |
|-------|-------|------------------|
| Admin-only signer | `initialize`, `request_randomness`, `close_game` | Non-admin creating/closing games or requesting randomness |
| VRF identity verification | `consume_randomness` | Anyone other than the VRF oracle injecting fake randomness |
| Not already revealed | `consume_randomness` | Double-consuming the VRF callback |
| Game not finished | `consume_randomness`, `guess` | Operating on a completed game |
| VRF consumed (is_revealed) | `guess` | Guessing before randomness is on-chain |
| Guess range [1, 100] | `guess` | Out-of-range guesses |
| Attempts remaining | `guess` | Exceeding max guesses (10) |
| Rejection sampling | `consume_randomness` | Modulo bias in secret number derivation |

The most critical check is the **VRF identity verification** in `consume_randomness`. Without it, anyone could call `consume_randomness` with their own randomness and control the secret number. The `#[account(address = ...)]` constraint ensures only the legitimate VRF program identity PDA can sign this instruction.

## Comparison: Phase 1 vs Phase 2 vs Phase 3

| Aspect | Phase 1 | Phase 2 (Switchboard) | Phase 3 (MagicBlock) |
|--------|---------|----------------------|---------------------|
| **Trust model** | Trust admin | Trustless (VRF) | Trustless (VRF) |
| **Randomness source** | Admin picks | Switchboard oracle | MagicBlock oracle |
| **Secret derivation** | Admin choice | `bytes[0] % 100 + 1` | Rejection sampling (unbiased) |
| **Cost** | ~0.00004 SOL | ~0.00006 SOL | **Free** |
| **Speed** | Instant | ~3 seconds | **<1 second** |
| **SDK** | blake3 | switchboard-on-demand | Manual (SDK incompatible) |
| **Account size** | 78 bytes | 118 bytes | **79 bytes** |
| **PDA seeds** | `[b"game", admin]` | `[b"game_v2", admin]` | `[b"game_v3", admin]` |
| **Instructions** | 4 (init, reveal, guess, close) | 4 (init, settle, guess, close) | 5 (init, request, consume, guess, close) |
| **VRF proof** | None | Switchboard commitment | RFC 9381 (Curve25519 Ristretto) |
| **Audit trail** | blake3 hash (admin) | VRF commitment | blake3 hash (from VRF bytes) |
| **Who knows secret** | Admin (before reveal) | Nobody (until settled) | Nobody (until callback) |

### When to use which

- **Phase 1 (Commit-Reveal):** Cheapest, fastest, simplest. Good when the admin is trusted (e.g., classroom demo, internal tool). No external dependencies.
- **Phase 2 (Switchboard VRF):** Provably fair randomness with a mature, battle-tested oracle. Worth the extra cost and latency for public-facing games where Switchboard infrastructure is already in use.
- **Phase 3 (MagicBlock VRF):** Best of both worlds — trustless AND free AND fast. The callback pattern is elegant but requires careful identity verification. Ideal when cost matters and sub-second latency is needed.

## Account Space Calculation

```
 8  discriminator          (Anchor internal)
32  admin                  (Pubkey)
32  secret_hash            ([u8; 32] -- blake3 audit trail)
 1  secret_number          (u8)
 1  is_revealed            (bool)
 1  attempts               (u8)
 1  max_tries              (u8)
 1  is_finished            (bool)
 1  bump                   (u8)
 1  vrf_request_pending    (bool)
---
79 total bytes
```

The program uses `#[derive(InitSpace)]` on the `GameV3` struct, so Anchor computes `GameV3::INIT_SPACE` automatically. The account is allocated with `space = 8 + GameV3::INIT_SPACE`.

**Comparison:** Phase 3 is 39 bytes smaller than Phase 2 (79 vs 118 bytes) because the callback pattern doesn't need to store `randomness_account` (Pubkey, 32 bytes) or `commit_slot` (u64, 8 bytes). The only extra field is `vrf_request_pending` (bool, 1 byte).

## What's Next

Phase 3 achieves free, fast, trustless randomness using MagicBlock VRF. Potential future improvements:

- **SDK migration:** Once `ephemeral-vrf-sdk` resolves the Anchor 1.0.1 `Pubkey` compatibility, replace manual CPI code with SDK calls.
- **Multiplayer mode:** Multiple players guess simultaneously, closest wins.
- **Staking:** Players stake SOL to enter, winner takes the pool.
- **Timed rounds:** Game auto-settles after a deadline.
- **Cross-program invocation:** Compose with other Solana programs (e.g., token rewards).
- **Phase 4:** Explore other VRF providers or commit-reveal schemes for comparison.
