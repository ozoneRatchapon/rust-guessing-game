# Phase 2: Switchboard VRF — Trustless On-Chain Randomness

## Overview

This phase implements a **trustless guessing game** on Solana using Switchboard's on-demand VRF (Verifiable Random Function). It is a **separate** Anchor program (`phase2-vrf`) from Phase 1 — both programs coexist on devnet with different Program IDs.

**How it works:**

1. The admin **initializes** a game. In the same transaction, the program commits to a Switchboard randomness request.
2. The Switchboard oracle generates randomness off-chain (~3 seconds).
3. The admin **settles** the randomness on-chain. The program derives the secret number from the VRF output.
4. Players **guess** the number. The program responds with "too small", "too big", or "correct".

**Trust model:** Nobody — including the admin — knows the secret number before it is settled. The secret is derived from a cryptographic randomness proof generated by the Switchboard oracle. This eliminates the Phase 1 limitation where the admin chose (and therefore knew) the answer.

**Key difference from Phase 1:** Phase 1 uses a commit-reveal scheme where the admin picks the number and commits its hash. Phase 2 replaces the admin-chosen number with a VRF oracle. The "commit" comes from the oracle's cryptographic commitment to the current slot hash, not from an admin transaction.

---

## Why a Separate Program (Not Replace Phase 1)

Phase 2 lives in its own program alongside Phase 1, not as a replacement:

1. **Phase 1 is already deployed and demo-able forever.** It works, it's tested, and it teaches the commit-reveal pattern.
2. **Phase 2 has different instructions.** No `reveal` instruction — instead `settle_random` for VRF settlement. The account struct is different too (VRF-specific fields).
3. **Better for teaching.** Students compare both approaches side by side: trust-on-admin vs. trustless VRF.
4. **Git tag `phase1-demo` preserves the Phase 1 snapshot.** You can always checkout the Phase 1 version.

Both programs run on devnet simultaneously with separate Program IDs. A frontend can integrate either or both.

## Architecture

### Phase 1 Flow (Trust-on-Admin)

```
initialize(secret_number) → reveal(secret_number) → guess(guess)
        |                          |                       |
  Admin picks number         Admin reveals number     Player guesses
  Stores blake3 hash         Verifies hash            Program compares
  Admin knows secret         Secret now public        Emits event
```

### Phase 2 Flow (Trustless VRF)

```
initialize → [wait ~3s for oracle] → settle_random → guess(guess)
     |                                       |                |
 Admin creates game                    Admin settles VRF   Player guesses
 Commits to Switchboard randomness     Derives secret      Program compares
 Nobody knows secret                   Secret now on-chain  Emits event
```

**Order matters:** `initialize` → wait for oracle → `settle_random` → `guess`. You cannot guess before the VRF is settled. The oracle wait (~3 seconds) is the trade-off for trustless randomness.

---

## Switchboard Commit-Reveal Pattern

Switchboard on-demand VRF uses a commit-reveal pattern, but the roles are different from Phase 1:

1. **Commit** (bundled with `initialize` in the same transaction) — The Switchboard program commits to the current slothash. This is the cryptographic "promise" that the randomness will be tied to a specific point in time. Nobody can predict what the slothash will produce.

   ```rust
   // In the same transaction as initialize:
   // [createIx, commitIx, initIx]
   // The commitIx tells Switchboard: "tie this randomness request to the current slot"
   ```

2. **Oracle generates randomness** off-chain — The Switchboard oracle observes the committed slothash and generates a provably random value. This takes ~3 seconds. Nobody can influence the result because the slothash was already committed.

3. **Reveal** (bundled with `settle_random` in the same transaction) — The Switchboard program reveals the randomness on-chain and verifies the proof. Our program reads the settled randomness and derives the secret number.

   ```rust
   // In the same transaction as settle_random:
   // [revealIx, settleIx]
   // The revealIx settles the randomness proof on-chain
   // The settleIx reads it and derives: secret = bytes[0] % 100 + 1
   ```

**Why this is trustless:** The randomness is derived from a slothash that didn't exist when the game was initialized. Even the admin cannot know what the slothash will be. The Switchboard oracle's proof is mathematically verifiable — if the proof is invalid, the transaction fails.

## Program: `phase2-vrf`

### Account Struct (GameV2)

The `GameV2` account stores all game state in a single PDA:

| Field              | Type      | Size | Description                                    |
|--------------------|-----------|------|------------------------------------------------|
| discriminator      |           | 8    | Anchor account discriminator                   |
| admin              | Pubkey    | 32   | Game administrator                             |
| secret_hash        | [u8; 32]  | 32   | VRF commitment (not admin-chosen)              |
| secret_number      | u8        | 1    | Derived from VRF after settle                  |
| is_revealed        | bool      | 1    | Whether VRF has settled                        |
| attempts           | u8        | 1    | Number of guesses made                         |
| max_tries          | u8        | 1    | Maximum guesses allowed (10)                   |
| is_finished        | bool      | 1    | Game over flag                                 |
| bump               | u8        | 1    | PDA bump seed                                  |
| randomness_account | Pubkey    | 32   | Switchboard randomness PDA                     |
| commit_slot        | u64       | 8    | Slot when randomness was committed             |
| **Total**          |           | **118** |                                               |

**Key difference from Phase 1:** Two extra fields — `randomness_account` (the Switchboard randomness PDA) and `commit_slot` (the slot when randomness was committed). These enable the trustless VRF flow.

### PDA Derivation

The game account is a Program Derived Address (PDA) derived from:

```rust
seeds = [b"game_v2", admin.key().as_ref()]
```

This uses `b"game_v2"` (not `b"game"`) to avoid PDA collisions with the Phase 1 program.

## Instructions

### initialize()

**Purpose:** Admin creates a new game and commits to Switchboard randomness. This instruction is bundled with the Switchboard `create` and `commit` instructions in the same transaction.

**Account validation:**

- `game` — initialized as a PDA with seeds `[b"game_v2", admin.key()]`, funded by the admin.
- `admin` — must be a signer and must pay for account rent.
- `randomness_account` — the Switchboard randomness PDA, passed as an account reference.
- `system_program` — required for account creation.

**What gets stored:**

```rust
game.admin = admin.key();
game.secret_hash = [0u8; 32];      // placeholder, filled after VRF settles
game.secret_number = 0;             // NOT known yet
game.is_revealed = false;           // VRF not settled
game.attempts = 0;
game.max_tries = MAX_TRIES;         // 10
game.is_finished = false;
game.bump = bump;
game.randomness_account = randomness_account.key();
game.commit_slot = current_slot;    // slot when randomness was committed
```

The `commit_slot` is critical for security — it records the slot at which the randomness was committed, allowing the program to verify freshness during settlement.

**Transaction bundling:**

```typescript
// Three instructions in one transaction:
const [randomness, createIx] = await sb.Randomness.create(sbProgram, rngKp, queue);
const commitIx = await randomness.commitIx(queue);
const initIx = program.methods.initialize().accounts({...}).instruction();

// Send: [createIx, commitIx, initIx]
```

---

### settle_random()

**Purpose:** Admin settles the VRF result on-chain and derives the secret number. This instruction is bundled with the Switchboard `reveal` instruction in the same transaction.

**Account validation:**

- `game` — mutable (we're writing the secret number).
- `admin` — must be a signer.
- `randomness_account` — the Switchboard randomness PDA, must match the stored reference.

**Security checks (in order):**

1. **Admin-only:** The signer must match the game's stored admin.

   ```rust
   require!(
       ctx.accounts.admin.key() == game.admin,
       GameError::Unauthorized
   );
   ```

2. **Not already revealed:** Prevents settling twice.

   ```rust
   require!(!game.is_revealed, GameError::AlreadyRevealed);
   ```

3. **Randomness account verification:** The passed randomness account must match what was stored at initialization.

   ```rust
   require!(
       ctx.accounts.randomness_account.key() == game.randomness_account,
       GameError::RandomnessAccountMismatch
   );
   ```

4. **Commit slot freshness:** The randomness commit slot must match the stored slot.

   ```rust
   require!(
       randomness.commit_slot == game.commit_slot,
       GameError::CommitSlotMismatch
   );
   ```

5. **Randomness settled:** The Switchboard randomness must have been resolved (status indicates success).

**Deriving the secret number:**

```rust
let randomness_bytes = randomness.value();  // 32 bytes of VRF output
game.secret_number = (randomness_bytes[0] % 100) + 1;  // 1-100
game.is_revealed = true;
```

The first byte of the VRF output is reduced modulo 100, then shifted to the 1-100 range. This is a uniform distribution over the game's range.

**Transaction bundling:**

```typescript
// Two instructions in one transaction:
const revealIx = await randomness.revealIx();
const settleIx = program.methods.settleRandom().accounts({...}).instruction();

// Send: [revealIx, settleIx]
```

---

### guess(guess)

**Purpose:** Any player can submit a guess. The program compares it to the secret and emits an event. This is identical to Phase 1's `guess` instruction.

**Account validation:**

- `game` — mutable (we're tracking attempts).
- `player` — must be a signer (any wallet, no restriction).

**Security checks (in order):**

1. **Game not finished:** Cannot guess on a completed game.

2. **Secret revealed (VRF settled):** Cannot guess before the VRF is settled.

3. **Attempts remaining:** Player has not exceeded `max_tries`.

**Comparison logic:**

```rust
match guess.cmp(&game.secret_number) {
    std::cmp::Ordering::Equal => {
        game.is_finished = true;
        emit!(GuessCorrect { guess, attempts: game.attempts });
    }
    std::cmp::Ordering::Less => {
        emit!(GuessTooSmall { guess, attempts: game.attempts });
    }
    std::cmp::Ordering::Greater => {
        emit!(GuessTooBig { guess, attempts: game.attempts });
    }
}
```

**Auto-finish:** After the guess, if `attempts >= max_tries` and the game isn't already finished, the game ends automatically.

---

### close_game()

**Purpose:** Admin closes the game account, recovering the rent-exempt lamports back to their wallet. Identical to Phase 1.

**Account validation:**

- `game` — mutable, closed via Anchor's `close` constraint. PDA seeds `[b"game_v2", admin.key()]` with bump.
- `admin` — must be a signer. Receives the recovered rent lamports.

**When to use:** After a game is finished and you want to start a new one at the same PDA.

### Instructions Summary

| Instruction     | Who    | Description                                          |
|-----------------|--------|------------------------------------------------------|
| `initialize()`  | Admin  | Creates game, commits to Switchboard randomness      |
| `settle_random()` | Admin | Reveals VRF result, derives secret from randomness  |
| `guess(guess)`  | Anyone | Submits guess, emits too small / too big / correct   |
| `close_game()`  | Admin  | Closes game account, recovers rent to admin          |

## Security Checks

The VRF flow introduces new attack vectors. Here are the security checks that prevent them:

| Check | Where | What it prevents |
|-------|-------|------------------|
| Admin-only signer | `settle_random`, `close_game` | Non-admin settling or closing |
| Not already revealed | `settle_random` | Double-settling the VRF |
| Randomness account match | `settle_random` | Swapping in a different randomness account |
| Commit slot match | `settle_random` | Using stale or manipulated randomness |
| Randomness settled status | `settle_random` | Settling before oracle has responded |
| Game not finished | `guess` | Guessing on a completed game |
| VRF settled | `guess` | Guessing before randomness is on-chain |
| Attempts remaining | `guess` | Exceeding max guesses |

## TypeScript Client Flow

The client script orchestrates the VRF flow using the Switchboard TypeScript SDK:

```typescript
import * as sb from "@switchboard-xyz/on-demand";

// 1. Initialize (commit randomness)
const queue = new sb.Queue(sbProgram, {
  pubkey: new PublicKey("FdRnYujMnYbAJp5P2rkEYZCbF2TKs2D2yXZ7MYq89Hms"),
});

const rngKp = Keypair.generate();
const [randomness, createIx] = await sb.Randomness.create(
  sbProgram,
  rngKp,
  queue
);
const commitIx = await randomness.commitIx(queue);
const initIx = program.methods
  .initialize()
  .accounts({
    game: gamePda,
    admin: adminPayer.publicKey,
    randomnessAccount: rngKp.publicKey,
    systemProgram: SystemProgram.programId,
  })
  .instruction();

// Send: [createIx, commitIx, initIx] in one transaction
const initTx = await sendTransaction(
  [createIx, commitIx, initIx],
  [adminPayer, rngKp]
);

// 2. Wait for oracle to generate randomness (~3 seconds)
await sleep(3000);

// 3. Settle randomness (reveal + settle in one tx)
const revealIx = await randomness.revealIx();
const settleIx = program.methods
  .settleRandom()
  .accounts({
    game: gamePda,
    admin: adminPayer.publicKey,
    randomnessAccount: rngKp.publicKey,
  })
  .instruction();

// Send: [revealIx, settleIx] in one transaction
const settleTx = await sendTransaction([revealIx, settleIx], [adminPayer]);

// 4. Guess loop (same as Phase 1)
// After settlement, the secret is on-chain and anyone can guess
const guessTx = await program.methods
  .guess(guess)
  .accounts({
    game: gamePda,
    player: playerPayer.publicKey,
  })
  .signers([playerPayer])
  .rpc();
```

## Comparison: Phase 1 vs Phase 2

| Aspect | Phase 1 | Phase 2 |
|--------|---------|---------|
| **Trust model** | Trust admin | Trustless (VRF) |
| **Secret source** | Admin picks | VRF oracle |
| **Instructions** | init, reveal, guess, close | init, settle_random, guess, close |
| **Who knows secret** | Admin (before reveal) | Nobody (until settled) |
| **On-chain cost** | ~40k lamports | ~50-60k lamports (extra VRF accounts) |
| **Setup time** | Instant | ~3 seconds (oracle wait) |
| **Account size** | 78 bytes | 118 bytes (extra VRF fields) |
| **PDA seeds** | `[b"game", admin]` | `[b"game_v2", admin]` |
| **Hash algorithm** | blake3 (admin commits) | VRF commitment (oracle commits) |
| **Security assumption** | Admin is honest | Oracle math is sound |

### When to use which

- **Phase 1 (Commit-Reveal):** Cheaper, faster, good enough when the admin is trusted (e.g., classroom demo, internal tool). Simpler to understand and implement.
- **Phase 2 (Switchboard VRF):** When you need provably fair randomness. Nobody — including the deployer — can influence or predict the outcome. Worth the extra cost and latency for any public-facing game or lottery.

## Account Space Calculation

```
 8  discriminator       (Anchor internal)
32  admin               (Pubkey)
32  secret_hash         ([u8; 32] -- VRF commitment)
 1  secret_number       (u8)
 1  is_revealed         (bool)
 1  attempts            (u8)
 1  max_tries           (u8)
 1  is_finished         (bool)
 1  bump                (u8)
32  randomness_account  (Pubkey -- Switchboard randomness PDA)
 8  commit_slot         (u64 -- slot when randomness was committed)
---
118 total bytes
```

The program uses `#[derive(InitSpace)]` on the `GameV2` struct, so Anchor computes `GameV2::INIT_SPACE` automatically. The account is allocated with `space = 8 + GameV2::INIT_SPACE`.

## What's Next

Phase 2 achieves trustless randomness using Switchboard VRF. Potential future improvements:
